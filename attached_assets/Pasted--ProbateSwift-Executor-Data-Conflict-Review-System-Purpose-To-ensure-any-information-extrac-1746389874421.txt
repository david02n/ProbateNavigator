📄 ProbateSwift – Executor & Data Conflict Review System
🎯 Purpose
To ensure any information extracted from uploaded documents that differs from existing executor or user data is not automatically overwritten. Instead, these conflicts are logged, presented to the user for manual confirmation, and persist in a reviewable state until resolved.

✅ Feature Overview
Whenever a user uploads a document (e.g. will, utility bill, ID document), our AI parsing system may extract:

Executor name(s)

Address

Email or phone number

Legal role or relationship

Date of birth or other identity details

If the extracted data differs from any field currently stored in the system (manually added or previously extracted), it should be treated as a conflict and handled as follows:

🔍 Conflict Detection Logic
Trigger:
After document upload and AI parsing

For any entity (executor, applicant, contact), compare each parsed field to the existing value in the database

If a difference is found:
Create a conflict entry

Do not update the original record

Notify the user that updates require their review

✍️ Manual Entry and Conflict Detection – Clarified Handling
✅ Manual Entry by the User
Users can always manually:

Add a new executor

Edit any existing executor fields (e.g. address, email, name)

Pre-fill executor info before uploading a will or receiving formal confirmation

🔁 How Manual Entry Interacts With Conflict Logic
1. Manual entries are treated as the “current truth”
Any manually entered data is stored as the primary value for that field

AI suggestions from later uploads are not allowed to overwrite this data directly

2. If a document is uploaded after manual entry
Any field differences between parsed AI values and manually entered values will be:

Logged as conflicts

Clearly labelled:
“The uploaded document suggests a different address for this executor”

User must manually accept or reject the suggested update

3. If a user manually edits a previously AI-extracted value
Manual edits overwrite the AI-suggested value

The conflict (if one was pending) should be marked as resolved by user entry

🧱 Summary: Conflict Priorities by Source
Source of Value	Priority Level	Overridable by AI?	Explanation
Manual Entry	✅ Highest	❌ No	Always treated as intentional and authoritative
AI-Extracted Value	Medium	❌ Only suggest, never replace	
Empty Field	Low	✅ Yes, if no user input exists and confidence is high	

🧑‍💻 Developer Notes on Manual Entry
Each field on an executor should include a source tag:

manual, ai_extracted, or blank

Conflict logic must only trigger when:

source = manual AND parsed value ≠ stored value

Manual edits after a conflict should:

Resolve the conflict

Mark it with resolved_by_user_entry: true

💡 UX Suggestion for Manual Entry
When the user manually edits a field with a pending conflict, show:

“A suggestion from your documents exists for this field. Your change will override it.”

Optionally allow users to view the suggestion before confirming their own input
