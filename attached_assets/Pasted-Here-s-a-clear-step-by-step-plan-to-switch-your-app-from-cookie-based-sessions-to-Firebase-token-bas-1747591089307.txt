Here's a clear step-by-step plan to switch your app from cookie-based sessions to Firebase token-based authentication using ID tokens, which will eliminate the 401 errors and make your app more secure and production-ready.

🔁 Goal:
Switch from relying on session cookies (e.g., express-session, probswft.sid) to using Firebase ID tokens sent in headers and verified by your backend.

✅ Migration Plan: Cookie → Token-Based Auth
1. [Frontend] Setup Firebase Auth (if not already done)
Ensure Firebase is initialized and the user can log in:

js
Copy
Edit
import { initializeApp } from 'firebase/app';
import { getAuth, GoogleAuthProvider, signInWithPopup } from 'firebase/auth';

const firebaseConfig = {
  apiKey: "YOUR_KEY",
  authDomain: "YOUR_PROJECT.firebaseapp.com",
  // ...other Firebase config values
};

const app = initializeApp(firebaseConfig);
const auth = getAuth(app);
const provider = new GoogleAuthProvider();
2. [Frontend] After login, get the ID token
Replace your current login handler with something like:

js
Copy
Edit
async function loginWithGoogle() {
  const result = await signInWithPopup(auth, provider);
  const user = result.user;
  const idToken = await user.getIdToken(); // 👈 this is what the backend needs

  return { user, idToken };
}
3. [Frontend] Send the token with your requests
Anywhere you make authenticated API calls, attach the token:

js
Copy
Edit
const { idToken } = await loginWithGoogle();

fetch('https://probateswift.com/api/user', {
  method: 'GET',
  headers: {
    'Authorization': `Bearer ${idToken}`
  }
});
🧠 You can store the token in memory, or call getIdToken() before each request.

5. [Backend] Create an authentication middleware
This replaces session checking with token verification:

js
Copy
Edit
// authMiddleware.js
const admin = require('./firebaseAdmin');

async function verifyFirebaseToken(req, res, next) {
  const header = req.headers.authorization;
  if (!header || !header.startsWith('Bearer ')) {
    return res.status(401).send({ error: 'Missing or malformed token' });
  }

  const idToken = header.split('Bearer ')[1];

  try {
    const decoded = await admin.auth().verifyIdToken(idToken);
    req.user = decoded; // 👈 you'll have uid, email, etc.
    next();
  } catch (err) {
    res.status(401).send({ error: 'Invalid token', details: err.message });
  }
}

module.exports = verifyFirebaseToken;
6. [Backend] Use the middleware in your routes
js
Copy
Edit
const express = require('express');
const verifyFirebaseToken = require('./authMiddleware');

const app = express();

app.get('/api/user', verifyFirebaseToken, (req, res) => {
  res.json({ user: req.user }); // or fetch user data from DB using req.user.uid
});

app.listen(3000, () => console.log('API running on 3000'));
7. [Frontend] Remove reliance on cookies or session
Remove:

express-session

req.session.user

Any assumptions about cookies containing auth state

If you need to persist login on refresh, Firebase handles this automatically via local storage.

🧪 Final Test
✅ User logs in via Google/Firebase

✅ You grab their ID token

✅ Your frontend sends it with all requests

✅ Your backend verifies it and returns protected data

✅ No 401 errors unless token is missing or expired

🧰 Bonus (Optional Enhancements)
🧠 Add a refreshIdTokenIfNeeded() method to get fresh tokens

🛡️ Add admin.auth().getUser(uid) in backend to enrich the user object

🪪 Optionally store a custom user profile in Firestore or your DB

✅ Summary
Step	Action
1	Use Firebase Auth on frontend
2	Get ID token with getIdToken()
3	Send it in Authorization header
4	Verify it with Firebase Admin SDK in Express
5	Protect API routes using middleware
6	Ditch cookies/sessions — you're stateless now
